# 非目標

本文件整理 `sdd-board` 在目前規劃階段**不打算優先處理**的方向，避免範圍膨脹。

## 核心原則

- 先把主流程做清楚
- 不過早把平台做成全能型系統
- 不在第一階段承擔過多外部平台責任

## 目前非目標

## 1. 取代 Jira / GitHub / GitLab / Bitbucket

`sdd-board` 的定位是平台內的發想、定案與流程串接中樞，  
不是要完全取代既有 ticket 或 repository 平台。

## 2. 取代 OpenSpec CLI 本身

目前方向是把 OpenSpec 的工作流程延伸到 Web，  
不是重寫整套 OpenSpec 規則引擎。

## 3. 把發想全自動化

AI 可以參與發想與提供輔助，  
但發想與定案本身不應被全自動取代。

## 4. 一開始就做成重型多服務架構

第一版優先輕量單體與模組化，  
不是一開始就拆成大量服務、佇列、分散式元件。

## 5. 一開始就支援所有 provider 的完整能力

ticket、repository、AI 都會保留可替換性，  
但第一階段不是一次把每一家 provider 的全部能力都做滿。

## 6. 一開始就做成完整企業治理平台

安全、權限、隔離會納入規劃，  
但第一版不是要一次實現所有大型企業級治理功能。

## 一句話總結

> `sdd-board` 的第一階段重點是把「project → room → decision → ticket」主流程做清楚，而不是同時取代所有外部系統或做成過重的平台。
