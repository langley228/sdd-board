# 自動化流程概覽

本文件整理 `sdd-board` 中自動化在整體流程中的位置與目前規劃方向。

## 核心定位

在 `sdd-board` 中，自動化不是發想的起點，而是：

```text
發想
  ↓
定案
  ↓
開票
  ↓
後續自動化
```

也就是說，自動化建立在：

- 已有 project 設定
- 已完成發想與定案
- 已產生 ticket

之後才開始接手後續流程。

## 自動化觸發點

### 1. 定案後

- 依 project 設定決定是否自動開票
- 若開票成功，則可進一步啟動後續自動化

### 2. ticket 建立後

- 可啟動 repository 綁定
- 可啟動後續 OpenSpec / Git 準備流程

### 3. repository 綁定後

- 可啟動 repository context 準備
- 可啟動後續自動驗證或預備流程

## 目前規劃的自動化類型

### Ticket 後流程

- 自動同步 ticket 狀態
- 自動更新 ticket 摘要或上下文

### Repository 後流程

- 自動準備 repository 上下文
- 自動檢查 default branch 與相關設定

### OpenSpec 後流程

- 自動執行後續 OpenSpec 驗證或準備動作

### AI 後流程

- 自動摘要定案內容
- 自動補齊後續流程所需文字內容

## 不應過早自動化的部分

- 發想本身不應被全自動取代
- 定案不應被自動視為完成
- 未經確認的內容不應直接進正式外部流程

## 與 project settings 的關係

自動化一定受 project settings 控制，例如：

- 是否允許自動開票
- 是否允許自動綁定 repository
- 是否允許自動執行後續流程
- 哪些 AI 能力可用於自動化

## 自動化原則

- 人工定案先於自動化
- 自動化只接手定案後流程
- 自動化的每一步都應可追蹤
- 自動化應可關閉、可配置、可擴充

## 一句話總結

> `sdd-board` 的自動化應定位為「定案後的流程推進層」，而不是取代發想與決策本身。
