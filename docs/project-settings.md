# 專案設定模型

本文件整理 `sdd-board` 中 project 作為流程起點時，應承接的設定面向。

## 核心概念

在 `sdd-board` 中，project 不只是資料分組單位，而是：

- 流程邊界
- 權限範圍
- 外部系統設定容器
- AI 與 automation 的策略來源

## 專案設定分類

```text
Project Settings
├─ Basic
├─ Ticket
├─ Repository
├─ AI
└─ Automation
```

## 1. Basic

### 目的

提供 project 的基本識別與操作上下文。

### 建議欄位

- `projectName`
- `projectKey`
- `description`
- `status`
- `owner`
- `defaultLanguage`

## 2. Ticket Settings

### 目的

定義定案後要如何開票與對應哪一個工作系統。

### 建議欄位

- `defaultTicketProvider`
- `ticketProjectKey`
- `ticketIssueType`
- `ticketTemplate`
- `statusMapping`
- `allowCreateTicket`

### 說明

- 支援 Jira、GitHub、GitLab 等 provider
- 可先保留單一預設 provider，後續再擴多 provider 策略

## 3. Repository Settings

### 目的

定義定案後要銜接哪一個或哪些儲存庫。

### 建議欄位

- `defaultRepositoryProvider`
- `repositories`
- `defaultBranch`
- `branchNamingRule`
- `allowGitFlow`
- `allowOpenSpecWrite`

### repositories 建議內容

每一個 repository 項目可包含：

- `provider`
- `name`
- `namespace`
- `repoUrl`
- `defaultBranch`
- `authRef`

### 第一版原則

- repository 設定在建立 project 時即先提供最小上下文
- 後續可在 project settings 中修改
- ticket 建立後直接沿用 project 的 repository context，不在 ticket 結果頁重新決定

## 4. AI Settings

### 目的

定義發想階段與後續自動化可使用的 AI 能力。

### 建議欄位

- `enabledProviders`
- `defaultChatProvider`
- `defaultGenerationProvider`
- `defaultAutocompleteProvider`
- `enabledModels`
- `defaultModelMapping`
- `modelMapping`
- `promptPolicy`
- `maskingPolicy`
- `usagePolicy`

### 說明

- project 可同時配置多個 AI provider
- 不同 AI 能力可對應不同 provider
- AI 設定需與安全邊界一起考慮
- project settings 負責定義 AI 的可用範圍與預設值
- 後續各階段仍可由使用者在 project 允許的範圍內，自行選擇要使用的 provider / model

## 5. Automation Settings

### 目的

控制定案後哪些流程要自動執行。

### 建議欄位

- `autoCreateTicket`
- `autoAssignRepository`
- `autoGenerateDrafts`
- `autoRunValidation`
- `autoSyncExternalStatus`
- `autoPrepareGitFlow`

## 設定原則

- project 是所有後續行為的策略來源
- 先以輕量設定為主，不一開始堆滿所有欄位
- 設定需保留後續擴充空間
- 外部 provider 必須以抽象欄位承接，不可寫死單一服務

## 一句話總結

> `sdd-board` 的 project settings 是整個平台的控制面，負責決定 ticket、repository、AI、automation 如何參與專案流程。
